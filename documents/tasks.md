# Tasks

## 目的

この文書は、`fake-openai-server` を次の方針に沿って
段階的に改善するための実装タスク一覧です。

- Python + FastAPI を用いた堅牢な API サーバーにする
- `uv` を中心に開発・実行・依存管理を統一する
- `ruff` と `ty` による静的品質管理を導入する
- 型、OpenAPI、コメント、テストを明示的に整備する
- DB を使わない stateless な構成を維持する
- `docker compose` を正規の起動・検証手段として扱う

## 前提

- DB は導入しない
- アプリケーション状態は永続化しない
- `docker compose` で embeddings サーバーと reranker サーバーを起動できることを維持する
- GPU 前提の compose 設定は意図的なものとして扱う
- AI エージェントは [AGENTS.md](AGENTS.md) に従って変更を行う

## タスク詳細

### Task 1: ディレクトリ構成の再整理

#### Task 1 の目的

単体スクリプト中心の構成から、
責務ごとに分離された保守しやすい Python パッケージ構成へ移行する。

#### Task 1 の作業内容

- Python パッケージ構成へ移行する
- 配置は `src/` ベースを第一候補とする
- embeddings と reranker の責務をモジュール単位で分離する
- API ルータ、スキーマ、サービス、設定、ログ設定を分ける
- エントリポイントを `uvicorn` から扱いやすいモジュールパスへ統一する
- 既存の `LoggerConfig.py` を含む共有処理を共通モジュールに集約する

#### Task 1 の完了条件

- 単一ファイルに API 定義、設定、モデル初期化、レスポンス構築が混在していない
- `docker compose` とローカル実行の両方で新構成が起動できる
- `README.md` の起動方法が新構成に追従している

### Task 2: `pyproject.toml` の整備

#### Task 2 の目的

開発ツール、テスト、型チェック、フォーマットをリポジトリ標準として宣言する。

#### Task 2 の作業内容

- Python バージョン制約を明示的に見直す
- 本番依存と開発依存を分離する
- `ruff` の設定を追加する
- `ty` の設定を追加する
- `pytest` と必要な補助ライブラリを開発依存へ追加する
- `tool` セクションに実行ルールを集約する

#### Task 2 の完了条件

- `uv sync` で開発に必要な依存がそろう
- `uv run ruff check .` が実行できる
- `uv run ty check` が実行できる
- `uv run pytest` が実行できる

### Task 3: 設定管理の型付き化

#### Task 3 の目的

環境変数ベースの設定を型安全にし、compose とコードの設定ズレを減らす。

#### Task 3 の作業内容

- `pydantic-settings` を導入する
- embeddings / reranker のモデル名、ホスト、ポート、ログ設定を型付き設定へ移行する
- `.env` と `compose.yaml` の変数定義を棚卸しする
- 起動時に必要な設定が不足している場合は明示的に失敗させる

#### Task 3 の完了条件

- 主要な環境変数を settings クラスで管理している
- `.env` と `compose.yaml` とコードの変数名が一致している
- 起動時の設定不備では、欠落または不正な環境変数名を含むエラーログを出して明示的に起動失敗する
- リクエスト処理中に設定不整合が検出された場合は、内部詳細を伏せた 5xx 系レスポンスとログ出力で扱う

### Task 4: FastAPI アプリケーション構成の見直し

#### Task 4 の目的

OpenAPI と運用性を意識した FastAPI 構成へ改める。

#### Task 4 の作業内容

- アプリケーションファクトリを導入する
- API ルータを分離する
- `response_model` を明示する
- OpenAPI のタイトル、説明、バージョン、タグを整備する
- ヘルスチェックエンドポイントを追加する
- `live` と `ready` の責務を分ける
- `/health/live` はプロセス生存確認、`/health/ready` は推論に必要な依存関係の初期化完了確認に限定する

#### Task 4 の完了条件

- `/openapi.json` の内容が主要 API を正しく表現している
- `/health/live` と `/health/ready` が利用できる
- embeddings と reranker の API が同じ設計ルールに沿っている

### Task 5: リクエスト・レスポンススキーマの明確化

#### Task 5 の目的

公開 API 契約を pydantic で表現し、型と OpenAPI の整合性を上げる。

#### Task 5 の作業内容

- embeddings の request/response モデルを定義する
- rerank の request/response モデルを定義する
- usage 情報や結果要素も個別モデルで表現する
- 必須項目、文字列/配列の型、空文字列・空配列の扱い、数値項目の範囲を明示する
- OpenAI 互換性を損なう独自制約は追加しない
- ad hoc な `dict` 返却を段階的に置き換える

#### Task 5 の完了条件

- 公開 API の主要な JSON 形状が pydantic モデルで表現されている
- OpenAPI 上で request と response のスキーマが確認できる
- 不正入力に対するバリデーション結果が一貫している

### Task 6: エラーハンドリングの標準化

#### Task 6 の目的

入力不正、設定不備、モデル推論失敗時のふるまいを統一する。

#### Task 6 の作業内容

- 共通エラーレスポンス形式を決める
- OpenAI API 互換のエラーレスポンス形式を基本とする
- FastAPI の例外ハンドラを整備する
- バリデーションエラー、内部エラー、依存初期化エラーの扱いを分ける
- ログ出力と HTTP レスポンスの責務を分離する

#### Task 6 の完了条件

- 主要な失敗パターンでレスポンス形式が統一されている
- 内部情報を過剰にレスポンスへ漏らさない
- 障害時にログだけ見れば原因を追いやすい

### Task 7: ロギングの整理

#### Task 7 の目的

コンテナ実行とローカル実行の両方で扱いやすいログ設計にする。

#### Task 7 の作業内容

- ログ設定を共有モジュールへ整理する
- 構造化ログは標準出力へ JSON 1 行 1 イベントで出力する
- ローカル開発でのみ任意にファイル出力を有効化できるようにする
- 起動ログ、リクエストログ、エラーログの粒度を整理する
- embeddings / rerank モデルへの入力件数、モデル名、処理時間、結果件数を `DEBUG` で記録する
- 生の入力本文、全文ベクトル、秘密情報、認証情報はデフォルトでは記録しない
- 機密情報や巨大 payload を不用意に出力しないようにする
- ログレベルごとの推奨出力項目を定義する

#### Task 7 のログレベル別推奨項目

- `DEBUG`
  - request_id / trace_id
  - エンドポイント名
  - 入力件数と正規化後メタデータ
  - モデル名
  - 推論オプション
  - 前処理・後処理の中間結果
  - 依存注入や設定解決の詳細
  - 実行時間の詳細内訳
- `INFO`
  - サービス起動・停止
  - モデルロード成功
  - リクエスト受付
  - レスポンス返却
  - HTTP メソッド
  - パス
  - ステータスコード
  - request_id / trace_id
  - 使用モデル名
  - 処理時間
  - クライアント識別に必要な最小情報
- `WARNING`
  - リトライ発生
  - 想定内だが注意が必要な入力
  - 一時的な依存不調
  - タイムアウト予兆
  - レート制限接近
  - フォールバック動作の発生
  - 非推奨 API 利用
- `ERROR`
  - リクエスト失敗
  - 推論失敗
  - 設定不備
  - 外部依存エラー
  - タイムアウト
  - 例外種別
  - request_id / trace_id
  - 失敗箇所
  - 再現に必要な最小限の入力要約
- `CRITICAL`
  - サービス継続不能な障害
  - 起動失敗
  - モデルロード不能
  - 致命的な設定不整合
  - 復旧に人手対応が必要な障害
  - 即時調査に必要な識別情報

#### Task 7 の補足方針

- `DEBUG` でも、入力本文そのものや埋め込みベクトル全量は原則として記録しない
- 調査に必要な場合は、件数、文字数、ハッシュ、先頭抜粋などの要約情報を優先する
- JSON 1 行につき 1 イベントとし、機械処理しやすいキー名を統一する

#### Task 7 の完了条件

- 両サービスで一貫したロギング方針になっている
- `docker compose logs` で必要情報を追跡できる
- JSON 形式の構造化ログを機械的に解析できる
- `debug` ログで推論処理のメタデータを追跡できる
- 通常運用で過剰ログにならない

### Task 8: モデル初期化と依存注入の整理

#### Task 8 の目的

モデルロードをテスト可能かつ安全にし、起動時失敗の扱いを明確にする。

#### Task 8 の作業内容

- SentenceTransformer / CrossEncoder の生成責務を分離する
- 依存注入またはラッパー経由で API 層から隠蔽する
- テスト時にモックへ差し替えられる構造へ変える
- 初期化戦略は起動時ロードに統一する
- `/health/ready` はモデルロード完了後にのみ success を返す
- モデルロード失敗時は起動失敗として扱い、失敗理由をログに残す

#### Task 8 の完了条件

- API テストで本物の重いモデルを必須にしない
- モデル初期化失敗時のふるまいが起動失敗として明示されている
- embeddings と reranker で初期化戦略が統一されている

### Task 9: コメント規約に沿った docstring 整備

#### Task 9 の目的

公開クラス・公開メソッドへ、保守可能な粒度のコメントを付与する。

#### Task 9 の作業内容

- 全ての公開クラスへ docstring を追加する
- 全ての公開関数・公開メソッドへ docstring を追加する
- コメント内容を「概要」「責務」「実装意図」「副作用」「例外条件」に絞る
- 低価値な逐語コメントは避ける

#### Task 9 の完了条件

- 公開 API や主要サービスの責務がコメントから短時間で理解できる
- コメントがコードの逐語説明になっていない
- 今後の AI エージェント実装時の判断材料になる
